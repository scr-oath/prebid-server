Sonic vs jsoniter — JSON benchmarks for prebid-server

Benchmark suite for issue #3437, comparing encoding/json, github.com/json-iterator/go, and github.com/bytedance/sonic on real prebid-server fixtures.

Why this matters: on the largest fixture (a 48 KB BidResponse), sonic/default is ~9× faster than today's jsoniter path on RoundTrip and uses ~half the allocations. On the smallest (188 B BidRequest) the gap shrinks to ~1.3×.

The bottom line: sonic is a real win on response-shaped payloads (large, deeply nested, lots of seatbids) and a marginal win on small request-shaped payloads. Whether that justifies migration cost depends on the answers in the decision report.

How to read this

Numbers below come from go test -bench=. -benchmem -benchtime=1s -count=2 on the fixtures committed under fixtures/. The median of the two runs is shown. Speedup is relative to jsoniter/validation-off (the lenient profile that drives most of prebid-server's adapter and exchange code via jsonutil.Unmarshal).

  • Unmarshal — bytes to Go struct (the entry-point path for /openrtb2/auction).
  • Marshal — Go struct to bytes (the response path, plus adapter request bodies).
  • RoundTrip — unmarshal then re-marshal back. Closest to the real per-request cost.

Allocations (lower is better)

Sonic's lower allocation count translates to less GC pressure under load. For the 48 KB response, sonic does ~15 allocs vs jsoniter's ~85 — that's a 5× reduction in garbage created per request.

Notes & caveats

  • Hardware: measurements were taken on Apple M4 Pro (arm64) — sonic now supports arm64 well, contradicting the "no ARM support" comment in issue #3437. Linux/x86_64 numbers will differ; the same harness can be re-run there.
  • Compiler version impact: sonic's JIT optimizations apply on Go 1.18+ for amd64 and 1.20+ for arm64. Anything below falls back to encoding/json.
  • The "merge-clone" path (the jsoniter extension prebid-server registers in util/jsonutil/merge.go) does not have a sonic equivalent. The bench includes it for fairness — if a sonic migration ships, that codepath would have to be re-implemented.
  • Custom encoders — prebid-server's RawMessageExtension compacts every json.RawMessage on encode; sonic doesn't expose jsoniter.ValEncoder, so the equivalent must be either upstream-compacted (one-time) or handled by replacing the type.
  • First-call cost: sonic's JIT pays a Pretouch penalty on the first encode/decode of an unseen type. The bench's ResetTimer() after a warm-up loop hides this; in production it's a 1-shot per type per process.
  • Variance: count=2 means the spread between min and max is reported in the JSON but not the tables. For a real go/no-go decision, run with count=10 on the target deployment platform.
